## 一、核心架构

### 1.1 四个报告的触发条件

| 时段 | 触发词 | 报告类型 |
|------|--------|---------|
| 盘前 9:00 | 盘前分析, 盘前, 隔夜, pre-market | 市场全景报告 |
| 集合竞价 9:25 | 竞价, 集合竞价, 竞价分析, 开盘前 | 竞价解读报告 |
| 早盘 10:00 | 早盘, 盘中, 盘中分析, 上午 | 盘中策略报告 |
| 复盘 15:00 | 复盘, 收盘, 收盘分析, 复盘报告, 盘后 | 收盘复盘报告 |
| 个股 (任意时间) | 分析股票, 个股分析, 研报, 股票报告, [代码] | 个股深度研报 |

当用户说"看看今天市场""今天怎么样"自动判断当前时间，选择最接近的报告时段。

### 1.2 时间判断逻辑

```
当前时间 < 9:15   → 盘前分析（含隔夜美股+昨日市场回顾）
9:15 ~ 9:40       → 集合竞价解读
9:40 ~ 11:35      → 早盘分析
11:35 ~ 14:00     → 午间休市提示（可简单概述上午情况）
14:00 ~ 16:00     → 收盘复盘
16:00 ~ 次日 9:00 → 盘前分析（含今日复盘+明日展望）
周末/节假日       → 周度回顾+下周前瞻
```

### 报告1：盘前分析 (9:00)

**采集内容：**

| 未来1-2月事件 | Brave search | 特定搜索未来IPO/会议/财报 |

**分析输出结构（⚠️ 盘前报告编写铁律 — 见下方说明）：**

### 🚨 盘前报告编写铁律（必读！）

**铁律1：隔夜美股数据必须核验**
- 道指/纳指/标普必须有具体点位和涨跌幅，不能只写方向
- 确认美股当天开市（搜索确认），休市日不可编数据
- 中概股必须列出代表个股涨跌幅（至少3只），不能笼统说'普涨/普跌'

**铁律2：情绪判断必须量化**
- 基于昨日涨跌比、涨停/跌停比、成交额变化、外盘表现四个维度加权
- 四个维度中至少有3个指向同一方向才能判定强势/恐慌，否则判定震荡/分歧
- 昨日上涨家数 > 3000 且 指数涨 > 1% → '强势'；反之上涨<1000 → '弱势'
- 涨停 > 跌停×3 → '强势'；跌停 > 涨停×3 → '恐慌'
- **结论必须前后一致，禁止既说'偏多'又说'谨慎'**

**铁律3：风险提示是必填模块**
- 以下条件触发必须警示：
  - 昨日涨停 > 60 → ⚠️ 过热警示
  - 隔夜美股跌 > 1.5% → ⚠️ 外盘拖累
  - 昨日指数涨但上涨家数<50% → ⚠️ 指数失真
  - 主力资金连续3日净流出 → ⚠️ 资金撤离
- 至少输出2条具体警示，每条必须有数据支撑

**铁律4：支撑压力区间必须有据**
- 压力位1 = max(昨日最高×1.005, 5日均线)
- 支撑位1 = min(昨日最低×0.995, 5日均线)
- 区间宽度应在[前5日ATR×0.5, 前5日ATR×1.5]范围内

**铁律5：数据来源必须明确**
- 美股数据标注来源（Reuters/Bloomberg/东方财富）
- 大宗商品价格标注来源和更新时间
- 避免'据传''业内人士'等模糊表述

**铁律6：产业大事件必须保留（防过度修正 🔥）**
- 删除不可靠数据时，有来源的产业大事件必须保留：SpaceX IPO/特斯拉FSD/人形机器人/千亿级产业基金等
- 每条大事件标注来源、最新进展、影响板块及具体受益A股标的（至少2只）
- 不因数据精度问题删除整条事件，标注'数据待确认'比直接删除好

**铁律7：板块分析必须给对标标的 🔥**
- 每个关注板块必须给至少2只受益A股标的（名称+代码）
- 标的必须与板块有直接业务关联，禁止笼统表述
- 示例：'存储芯片→深科技(000021)、北京君正(300223)'

**铁律8：跨报告连贯性 🔥**
- 盘前报告要引用昨日复盘的核心预判
- 如'昨日复盘判断半导体获利回吐，隔夜美光大涨19%可能改变这一格局'
- 盘前→竞价→早盘形成推演链条，不能各说各话

  ③ 主力资金净流入但超大单在流出（资金背离）
- 诱空判定需要同时满足以下 ≥2条：
  ① 跳水幅度>2%但量能未异常放大（量比<1.2）
  ② 有明确大单承接（逐笔成交可见百万级买单）
  ③ 抛压迅速衰减（5分钟内跌幅收窄>50%）
- 不足2条时判定为'真实方向'

**铁律2：资金流向不能只看流入榜**
- 必须同时展示流入TOP3和流出TOP3
- 流入流出比值<1.5时判定为资金分歧
- 板块轮动判断: 资金从前一交易日领涨板块流出→轮动信号

**铁律3：量价配合是核心**
- 上涨必须放量(量比>1.0)才可信，缩量上涨→回调风险
- 下跌时量比>1.5且放量下跌→抛压加重
- 半小时成交额/全日预期=放量/缩量的量化判断

**铁律4：风险提示必须结合半小时数据**
- 至少输出1条基于早盘数据的风险警示
- 不能只说'关注风险'，必须指出具体风险点

**铁律4b：北向/南向资金 + 两市量能必填 🔥**
- 必须有北向资金早盘流向（沪股通+深股通分别列出）
- 必须有南向资金早盘流向
- 必须有半小时两市成交额及同比变化
- 以上数据缺失时须标注'数据待获取'，不能跳过

**铁律4c：创业板/深成指数据精度 🔥**
- 创业板和深成指数据必须用腾讯API实时行情核验（小数点后两位）
- 两个指数与上证指数必须是同一时间戳的数据
- 偏差>0.1%则重新获取

**铁律5：跨报告数据一致性（新增 🔥）**
- 早盘报告引用昨日数据时，必须与上一份报告（竞价报告/复盘报告）核对，数据不一致时必须说明差异原因

**铁律6：板块涨幅必须检查是否由个股带动（新增 🔥）**

## 🚨 所有报告通用铁律（防过度修正 + 连贯性 + 专属风险 🔥）

### 通用铁律1：不准因怕犯错而过度删减 🔥
- 有来源的产业大事件（SpaceX IPO/特斯拉FSD/人形机器人/千亿产业基金等）必须保留，只删除无来源的编造数据
- 宁可通过`web_search`补充事件细节，也不整体删除事件模块
- 每条事件必须标注来源+最新进展+影响板块+具体受益标的

### 通用铁律2：跨报告连贯推演 🔥
- 盘前报告必须在开头引用昨日复盘的关键结论
- 竞价报告必须引用盘前报告预判并说明对错
- 早盘报告必须引用竞价报告的预判并验证
- 示例格式：'上一份报告预判半导体主线延续，实际竞价半导体+2.67%验证该判断'
- 三份报告形成：盘前提框架→竞价定方向→早盘做验证 的推演链条

### 通用铁律3：专属风险警示，禁止套话 🔥
- 每天写1-2条当日最大风险，格式：'⚠️ 今日最大风险：XXXX'
- 禁止使用'股市有风险，投资需谨慎'等套话

  - 昨日涨停>60且指数微涨 → ⚠️ 过热+指数失真

### 🚨 复盘报告编写铁律（必读！）

**铁律1：情绪判断必须是数据说话**
- 上涨家数占比 > 65% → 赚钱效应明显 ✅
- 上涨家数占比 45%~65% → 赚钱效应一般 ➖
- 上涨家数占比 < 45% → 亏钱效应明显 ❌
- 涨停/跌停比 > 3:1 → 情绪强势；< 1:2 → 情绪恐慌
- **涨跌家数和涨停跌停必须同时纳入情绪判断，不能只看指数涨跌**
- 指数涨但上涨<45% → 强制判定'指数失真'，必须标注❗

**铁律2：板块分析必须横向对比**
- 展示今日涨跌幅同时，必须标注昨日该板块的涨跌幅
- 连续3日上涨的板块 → 标注'连续走强' / '过热需警惕'（根据涨幅大小）
- 今日领涨板块如果昨日在跌幅榜 → 标注'轮动反弹'
- 今日领跌板块如果昨日在涨幅榜 → 标注'获利回吐'

**铁律3：个股深度分析不能只有一句结论**
- TOP3个股必须回答：①驱动因素（消息/基本面/板块/资金） ②资金性质（游资/机构/散户） ③持续性判断
- 涨幅>7%但主力净流入为负 → 标注⚠️ 资金背离
- 涨停股封单/成交额<0.5 → 标注'封板力度弱'

**铁律4：风险提示必须结合当日数据**
- 至少2条具体警示，每条必须有当日数据支撑
- 不能只说'股市有风险'

**铁律5：下午变化是精华，不能敷衍**
- 必须分析14:00-15:00的走势和原因，不能只写一句话
- 如果是尾盘拉升/跳水 → 标注是否伴随放量

**铁律6：数据口径统一+交叉核对（新增 🔥）**
- 同一主体使用统一数据口径：半导体=申万二级半导体板块
- 无法统一口径时：标注'半导体(申万二级)：X亿' vs '电子(申万一级)：Y亿'
- 资金数据标签规则：所有资金数据必须标注'主力资金（特大单+大单）'
- ⚠️ 各板块资金数据不得混淆：写半导体就用半导体的253亿，不可误用通信的64亿
- 指数名称统一规则：市场分析主指数强制用'科创50'，禁止使用'科创综指'
- 报告生成后逐项自查：同一数据是否在不同位置出现矛盾

**铁律7：量能定性必须准确（新增 🔥）**
- 成交额同比>0 且 指数上涨 → 放量上涨
- 成交额同比>0 且 指数下跌 → 放量下跌
- 成交额同比<0 且 指数上涨 → 缩量上涨
- 成交额同比<0 且 指数下跌 → 缩量下跌
- 禁止出现'放量374亿但判定为缩量'这种自相矛盾

**铁律8：极端分化行情特殊处理（新增 🔥）**

- 上涨家数占比<30% 且 涨停>跌停 → 判定为'极端撕裂'行情
  特征：'权重护盘、小票暴跌、虚假修复'
- 涨跌比和涨跌停比矛盾时：涨跌比优先级 > 涨跌停比
  即超4000股下跌时，即使涨停多于跌停也判定亏钱效应极差
- 指数红但涨跌比<30% → 强制标注❗'虚假修复：指数翻红但九成个股下跌'

**铁律9：护盘vs企稳必须区分（新增 🔥）**
- 尾盘翻红但全市场上涨家数<50% → 判定为'权重护盘'
- 尾盘翻红且全市场上涨家数>50% → 判定为'真实企稳'
- 护盘特征：涨幅集中在权重股/ETF，小票无反弹
- 企稳特征：板块普涨，中小盘跟涨
- 判定为护盘时禁止使用'探底回升'等暗示反转的措辞

**铁律10：收盘价在区间中的位置必须评估（新增 🔥）**
- 收盘价在核心区间上1/3 → 偏强，倾向压力位突破
- 收盘价在核心区间中1/3 → 方向不明，不给出倾向性判断
- 收盘价在核心区间下1/3 → 偏弱，倾向支撑位测试
- 收盘价卡在区间正中时：标注'方向不明，等待次日确认'

**铁律11：核心矛盾板块数据精确核对（新增 🔥）**
- 当日涨跌幅TOP3板块、核心矛盾板块的资金数据必须精确到亿
- 板块资金数据摘取时，必须保留原数据源的板块名标注
- 核心矛盾板块的资金数据偏差超过10%即判定为数据质量事故

**铁律12：指数名称统一规则（新增 🔥）**
- 全市场分析统一使用以下主指数：上证指数、深证成指、创业板指、科创50
- 禁止使用科创综指替代科创50

## 四-A、防幻觉与数据质量铁律（必读）

> ⚠️ **本技能所有数据输出必须经过浏览器核验。** 以下是不可违反的铁律。

### 铁律1：看见标签≠看见数据

浏览器页面上出现"韩国KOSPI""日经225"等标签，**不等于**这些市场今天有交易数据。
- 如果页面没有显示实际价格数字 → 说明市场休市 → **绝对不能自己编一个数字**
- 必须确认数据确实加载出来后，才能引用

### 铁律2：每个数字必须有来源

报告中的每个指数涨跌幅、涨停数、资金金额、公司估值，必须能追溯到：
- 浏览器截图上的可见数字，或
- web_search 返回的明确数据

**不允许：** 模糊估计、"大概""预估"、看到标签就脑补数字

### 铁律3：市场日历先行

引用任何境外市场（美股、日股、韩股、港股等）前，先确认今天是否开市：
```
web_search搜索："韩国 股市 今天 休市 2026年5月XX日"
```
只有确认开市后才能引用该市场数据。

### 铁律4：数据一致性检查

报告写完后，通读一遍检查：
- 数字合理吗？（休市日不可能有波动）
- 数量级对吗？（15万亿 vs 1.75万亿）
- 前后矛盾吗？（涨停数写6家但下文提到8只个股）

### 铁律5：宁缺毋滥

拿不准的数据，**宁可不写也不要编**。
- 可以写"数据待核实"
- 可以省略不确定的数据点
- 编造的数据会毁了整份报告的可信度

## 五、交叉验证机制

### 5.1 数据验证等级

```
等级A（高可信）：以下数据来源互相一致
- stock_data.py AKShare 输出 vs 浏览器东方财富页面
- 多个搜索结果的相同数据

等级B（中等）：2个独立来源一致
- 搜索结果数量≥2且数据一致
- 或 单个可靠来源（东方财富/同花顺）获取

等级C（低可信/需标注）：仅1个来源数据
- 必须标注"来源：{来源}，可能不完整"
- 带有"据传""疑似"等不确定性标签
```

### 5.2 数据核验流程

```
1. 行情数据核验
   stock_data.py indexes/updown/limits → 浏览器东方财富开盘啦验证

2. 板块数据核验
   collector.py sector → 浏览器东方财富板块资金页验证
   不一致时：以浏览器页面为准，标注stock_data偏差

3. 个股价格核验
   至少2个来源：东方财富 + 同花顺/腾讯
   不一致超过1%时：search/refresh，标注来源

4. 财务数据核验
   搜索得到财报摘要 → 如有官网财报链接则核实
   注意报告期（季度/半年/全年）

5. 新闻/消息核验
   至少2个独立来源一致才确认
   单一来源消息标注"引用自{来源}"
```

### 5.3 验证系统（v2.0 已升级 — 无法伪造）

**核心改动：核验器现在会主动抓取真实数据，不再是文件签字游戏。**

```bash

# 查看核验状态

# 数据自洽性检查

# 交叉验证（Browser + API）

# 生成验证报告

```

**旧版问题修复：**
- 旧版只检查 `/tmp/a_share_browser_verify.json` 文件是否存在（模型可伪造）
- v2.0 执行 verify 时真正调用 API 抓取指数和板块数据，存入核验记录
- 核验记录包含实际数据，email 发送前必须通过此检查

### 11.4 学习能力详解

#### ① 板块资金流模式学习
- **问题**：哪些板块资金流入后次日持续性强？
- **方法**：统计每个板块在历史中"资金流入日→次日继续涨"的概率
- **输出**：板块连续性排行（如"半导体连续性75%→追高安全""房地产连续性20%→一日游"）

#### ② 涨停密度分析
- **问题**：涨停多少决定次日情绪？
- **方法**：按涨停数区间统计次日上涨概率
  - 涨停<20家 → 次日表现如何？
  - 涨停60-80家 → 次日表现如何？
  - 涨停/跌停比>3 → 次日持续偏多概率？
- **输出**：涨停密度→次日概率映射

#### ③ 板块关联性学习
- **问题**：A板块涨了→B板块次日是否跟涨？
- **方法**：遍历所有板块对，统计"A涨→B次日也涨"的概率
- **输出**：板块传导链（如"半导体涨→科创50涨概率80%""新能源涨→锂电涨概率75%"）

#### ④ 成交量模式学习
- **问题**：放量/缩量后次日怎么走？
- **方法**：对比放量>15%和缩量>15%后的次日大盘表现
- **输出**：放量次日上涨概率、缩量次日上涨概率

#### ⑤ 星期效应学习
- **问题**：周几最容易涨？
- **方法**：统计每个工作日的历史平均涨跌幅和上涨概率
- **输出**：各工作日表现统计（如"周五上涨概率65%，周四最弱"）

#### ⑥ 历史相似日匹配
- **问题**：今天和历史上哪天最像？那天之后怎么走的？
- **方法**：多维相似度算法（涨跌比、涨停数、成交量、板块结构、指数涨跌幅 加权匹配）
- **输出**：TOP5相似日 + 各相似日次日表现

#### ⑦ 异常信号检测
- **问题**：今天有什么不正常的？
- **检测项**：
  - 涨停数偏离近期均值2个标准差以上
  - 成交量异常放大/缩小
  - 板块涨跌幅极端（>5%）
- **输出**：异常清单+严重程度

### 11.9 数据质量与注意事项

1. **数据积累需要时间**：前10个交易日的学习结果仅供参考，30个交易日以上开始有统计意义
2. **规则不是预言**：学习到的规则是概率性的（如"板块连续性75%"意味着10次中有7-8次对），不能视为绝对真理
3. **市场会变风格**：过去有效的模式可能在风格切换后失效，系统会持续跟踪并调整权重
4. **自动验证保守**：review_outcomes 命令仅验证信号日的次日大盘涨跌，不验证个股；复杂策略建议人工核对
5. **数据文件保护**：`data/trading_memory.json` 是学习的核心资产，建议定期备份

## 👥 十二、三身份协作系统（v1.3 新增）

> 小A·小C·小V — 三人各司其职，从数据到分析到实现到输出，全链路打通。

三位不是分析师，而是三种不同的「能力角色」，可以嵌入到任何工作流中：

| 角色 | 名字 | 身份 | 信条 | 做什么 |
|:----:|:----:|:----:|:----:|:-------|
| 🅰️ | 小A | 数据分析师 | 数据会说话，关键在于你会不会听 | 收集信息、分析数据、输出结论 |
| 🅲 | 小C | 程序员 | 代码不会撒谎，但bug会 | 写代码、建工具、自动化 |
| 🆅 | 小V | 自媒体运营 | 好的内容自己会传播 | 做图文、短视频、专业包装 |

### 12.1 协作流水线

```
  📡 原始数据
     │
     ▼
  🅰️ 小A（数据分析师）
     │  → 采集分析 → 结构化结论
     │
     ├──→ 🅲 小C（程序员）
     │       → 代码实现 → 自动化工具
     │
     └──→ 🆅 小V（自媒体）
             → 视觉包装 → 可传播内容
     │
     ▼
  📊 最终输出（报告/图表/视频）
```

### 12.2 角色能力一览

#### 🅰️ 小A — 数据分析师

**可用技能：**
- `a-share-master` — A股市场分析
- `a-stock-analysis-pro` — 个股深度研报
- `stock-analysis-agent` — 全球股市分析
- `financial-report` — 财务报告生成
- 多源搜索 — web_search / tavily-search / brave-search

**工作流程：**
1. 收集原始数据（行情/板块/资金/消息）
2. 分析市场结构和情绪
3. 输出结构化分析结论和策略建议
4. 将结论传递给小C（实现）和小V（包装）

**协作方式：**
- 给C：提供需要实现的算法逻辑和计算需求
- 给V：提供分析结论、关键数据点、图表需求

---

#### 🅲 小C — 程序员

**可用技能：**
- `auto-coder` — 自动代码生成与修复
- `web-generator` — 网页/工具开发
- `diagram-maker` — 架构图/流程图制作
- `skill-creator` — 技能创建与优化

**工作流程：**
1. 接收小A的算法需求
2. 实现代码/脚本/工具
3. 运行测试验证正确性
4. 修复bug直到通过
5. 为小V的输出提供数据接口

**协作方式：**
- 从A接收：分析算法、计算逻辑、策略规则
- 给V提供：数据导出接口、生成的图表HTML、自动化脚本

---

#### 🆅 小V — 自媒体运营

**可用技能：**
- `ai-image-generation` — AI图像生成（FLUX/SD/OpenAI）
- `ai-video-generation` — AI视频生成（Kling/Seedance/Wan）
- `video-studio` — AI影视工作室
- `meme-maker` — 梗图制作
- `canvas` — HTML画布呈现
- `compdf-conversion-cli` — PDF/图片格式转换

**工作流程：**
1. 接收小A的分析结论和小C的数据接口
2. 设计视觉呈现方案
3. 制作HTML报告/信息图/短视频
4. 输出最终可传播的内容

**协作方式：**
- 从A接收：分析结论、关键数据、策略建议
- 从C接收：数据接口、生成的图表代码、自动化脚本

### 12.3 启动协作

```bash
# 查看角色档案
python scripts/analysts.py role          # 列出所有角色
python scripts/analysts.py role A        # 查看小A的详细档案
python scripts/analysts.py role C        # 查看小C的详细档案
python scripts/analysts.py role V        # 查看小V的详细档案

# 启动协作流程
python scripts/analysts.py run           # 展示每日协作流水线
python scripts/analysts.py run "个股深度分析+研报生成"  # 自定义任务

# 生成子agent配置（可在对话中启动）
python scripts/analysts.py spawn A "分析今日市场数据"
python scripts/analysts.py spawn C "实现板块轮动追踪脚本"
python scripts/analysts.py spawn V "制作今日复盘HTML报告"

# 每日全流程
python scripts/analysts.py daily
```

### 12.4 协作示例：一个完整的交易日

假设今天收盘后，完整的三身份协作流程：

```
🅰️ 小A → 调用web_search采集收盘数据
        → 运行trading_memory.py记录快照
        → 运行strategy_learner.py学习模式
        → 输出分析结论（板块资金流向/情绪判断/明日策略）
        → 把结论传给小C和小V

🅲 小C → 收到小A的策略建议
        → 调用auto-coder实现新的计算指标
        → 运行测试确保代码正确
        → 为小V提供数据接口

🆅 小V → 收到小A的分析结论和小C的数据
        → 用canvas制作HTML报告
        → 用ai-image-generation生成配图
        → 产出最终可传播的复盘报告
```


### 12.6 在对话中启动子agent

在 OpenClaw 对话中，可以直接通过 spawn 命令让某个角色工作：

```
# 格式
sessions_spawn taskName="role_c" task="..." mode=run

# 实际例子 — 让小C写一个板块轮动分析脚本
sessions_spawn
  taskName: role_c
  task: |
    你正在以 🅲 小C（程序员）的身份执行任务。
    请实现一个Python脚本，功能：
    1. 读取trading_memory.json中的板块数据
    2. 计算每日板块轮动速度
    3. 输出板块持续性排名
    4. 运行测试验证
  mode: run
```
